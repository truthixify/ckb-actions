# CLAUDE.md

Instructions for Claude when working on this codebase.

## Project

CKB Action Links: a protocol and reference implementation for sharing CKB transactions as URLs. See `spec.md` for the protocol. This repo contains:

- **`packages/sdk`**: TypeScript SDK for building Action Endpoints and Clients.
- **`packages/client`**: Reference web Client that parses Action URLs, renders previews, and invokes wallets via CCC.
- **`packages/server`**: Reference Action Endpoint server with example actions.
- **`examples/`**: Working example actions (tip jar, DOB mint, invoice).

## Stack

Use these. Do not substitute without asking.

| Component | Choice |
|---|---|
| Language | TypeScript (strict mode) |
| Server | Express |
| Client | React + Vite |

## Setup before any work

If the repo does not yet exist, initialize it:

- Run `git init` and create a sensible `.gitignore` for Node/TypeScript projects.
- Set up `pnpm` workspaces. Use `pnpm`, not `npm` or `yarn`.
- Add a root `tsconfig.json` with strict mode enabled. Packages extend it.
- Add ESLint and Prettier with sensible defaults at the root.
- Add Vitest for testing.
- Make an initial commit titled `chore: initial repo setup`.

## Git workflow

- Commit early, commit often. Each commit should represent one logical change.
- Use Conventional Commits format: `type(scope): description`. Types: `feat`, `fix`, `chore`, `refactor`, `docs`, `test`, `style`.
- Do not add a `Co-Authored-By` line. Do not mention Claude in commit messages. Commits are authored by the repo owner.
- Do not add "Generated by Claude" or similar footers.
- Push to remote only when a coherent batch is done. A batch is one of:
  - A feature complete enough to be usable (e.g. "manifest schema validation works end to end").
  - A meaningful refactor that leaves the tree in a working state.
  - A bug fix with its regression test.
- Never push broken code. Run the test suite locally before pushing.
- Never force push to `main`.

## Code quality bar

Write code as if a senior engineer will review every line.

- TypeScript strict mode. No `any` without an explicit justification comment.
- All exported functions and types have JSDoc comments describing what they do, not how.
- Validate all external input at boundaries (HTTP request bodies, URL params, env vars). Use Zod.
- Errors are typed. Never throw raw strings. Use custom error classes with discriminated tags.
- No magic numbers or strings. Hoist to named constants.
- No dead code. If something is unused, delete it. Git remembers.
- Functions do one thing. If a function name needs `and` in it, split it.
- Prefer pure functions. Side effects are pushed to the edges.
- Tests cover the public API of each package. Aim for behavior coverage, not line coverage.

## Comment style

- Comments explain *why*, not *what*. The code shows what.
- Do not number steps in comments using `1.`, `2.`, `3.`. If a process has steps, use prose or labeled blocks:

  ```ts
  // bad
  // 1. Fetch the manifest
  // 2. Validate the schema
  // 3. Render the card

  // good
  // Fetch and validate the manifest, then render the card.
  ```

- Avoid TODO comments in committed code. If something is unfinished, open a GitHub issue and reference it.
- No commented-out code. Delete it.

## Architecture conventions

- Domain logic lives in `packages/sdk`. Server and client are thin wrappers over the SDK.
- The Manifest schema and OTX payload format are the source of truth. Define them once in the SDK with Zod, derive types, import everywhere.
- HTTP concerns (Express handlers, fetch calls) never appear in domain code.
- Wallet integration goes through CCC. Do not call wallet-specific APIs directly.
- All async functions handle their own errors at the appropriate boundary. Do not let promise rejections bubble unhandled.

## Server conventions (Express)

- Use the latest Express major (v5+).
- Use `express.json()` for body parsing. No `body-parser` package directly.
- One route file per logical resource. Mount at the root in `app.ts`.
- Middleware stack: request id, structured logger, CORS, JSON body parser, route handlers, error handler last.
- Use a centralized error handler middleware. Translate SDK errors into HTTP responses there.
- All handlers are async. Wrap with an `asyncHandler` helper so errors flow to the error middleware.
- Validate request bodies and params with Zod at the top of each handler.

## Client conventions (React + Vite)

- React 18+, function components only. No class components.
- Hooks for state. Zustand for any cross-component state, not Redux.
- Routing with `react-router` if needed. Most flows are single-page.
- Tailwind for styling. No CSS-in-JS.
- Use `@ckb-ccc/connector-react` for wallet UI. Do not roll your own.
- Fetch with native `fetch`. No axios.

## Naming

- Files: kebab-case (`action-manifest.ts`).
- Types and classes: PascalCase (`ActionManifest`).
- Functions and variables: camelCase (`parseManifest`).
- Constants: SCREAMING_SNAKE_CASE only for truly constant values (env keys, magic strings from external specs). Otherwise camelCase.
- Boolean variables and functions: prefix with `is`, `has`, `should`, `can`.

## Testing

- Each package has its own test suite under `<package>/tests/`.
- Test names describe the behavior, not the implementation: `returns 400 when address is malformed`, not `validateAddress fails`.
- Use Vitest. No Jest.
- Integration tests for the server use supertest against the Express app.
- Do not mock things that are cheap to use for real (a parser, a validator). Mock only network and time.

## Spec compliance

The protocol is defined in `spec.md`. The code must conform.

- When implementing a section, reference it in a JSDoc tag: `@spec §6.2`.
- If the spec is ambiguous, do not guess. Open a question in `spec.md` under "Open questions" and ask before coding.
- Spec changes are commits in their own right, tagged `docs(spec): ...`.

## What not to do

- Do not introduce new dependencies without justification. Each new package in `package.json` should earn its place.
- Do not ship code that has not been run.
- Do not write speculative abstractions. Build the concrete thing first. Refactor when a second use case appears.
- Do not silently catch errors. If you catch, you handle.
- Do not commit secrets, even to `.env.example`. Use placeholder values.

## When in doubt

- Read `spec.md`.
- Read the CCC docs at https://github.com/ckb-devrel/ccc.
- Read the CKB CoBuild RFC at https://talk.nervos.org/t/ckb-open-transaction-otx-cobuild-protocol-overview/7739.
- If still unclear, ask before writing code.
